# 前端部署文档

本文档说明 `flow-talk-web` 的生产部署方式，目标是：前端由 Nginx 托管静态文件，后端由同一个 Nginx 反向代理，浏览器端统一通过同域 `/api` 和 `/api/ws` 访问后端。

## 1. 部署结论

当前项目默认是 React Router SSR 构建：

```ts
// react-router.config.ts
export default {
  ssr: true
};
```

如果服务器不启动前端 Node 服务，只用 Nginx 托管静态文件，需要先切换成 SPA 构建：

```ts
// react-router.config.ts
export default {
  ssr: false
};
```

同时需要确认生产环境不要把接口地址构建成固定后端 IP。前端请求应保持同域：

```text
/api/...
ws(s)://当前域名/api/ws
```

> 注意：当前 `app/request/index.ts` 的默认值是 `http://127.0.0.1:8080`。如果要使用本文档的同域部署方式，需要把生产环境的 API 基准地址调整为当前站点同源，否则浏览器会继续请求本机 `127.0.0.1:8080`。

## 2. 构建前检查

检查 Node 版本：

```bash
node -v
```

React Router 8 当前构建提示要求 Node 版本大于 `22.22.0`。如果低于该版本，可能暂时还能构建通过，但不建议用于生产构建环境。

安装依赖：

```bash
npm install
```

类型检查：

```bash
npm run typecheck
```

生产构建：

```bash
npm run build
```

纯 Nginx 静态部署需要确认构建产物中存在入口文件：

```text
build/client/index.html
```

如果 `build/client/index.html` 不存在，通常说明还在 SSR 构建模式，不能直接把 `build/client` 当成完整站点托管。

当前项目在 `ssr: true` 下构建会生成：

```text
build/server/index.js
```

这表示当前产物需要 Node 服务运行前端 SSR。如果目标是纯 Nginx 静态部署，应先切到 `ssr: false` 后重新构建。

## 3. 前端接口地址

生产环境推荐使用同域代理，不在前端产物中写死后端地址。

正确形式：

```text
/api/auth/login
/api/users
/api/ws
```

不推荐形式：

```text
http://服务器IP:8080/api/...
ws://服务器IP:8080/api/ws
```

这样做的好处是：

- 避免浏览器跨域配置。
- HTTPS 站点下不会出现 mixed content。
- 后端真实地址只暴露在 Nginx 配置中，前端构建产物不需要变更。
- WebSocket 可以跟随页面协议自动使用 `ws://` 或 `wss://`。

推荐把 API 基准地址改成浏览器当前站点 origin，保留 `VITE_API_BASE_URL` 作为特殊环境覆盖：

```ts
const API_BASE_URL = import.meta.env.VITE_API_BASE_URL ||
  (typeof window !== "undefined" ? window.location.origin : "http://127.0.0.1:8080");
```

不要把 `VITE_API_BASE_URL` 配成空字符串；当前代码使用 `||`，空字符串仍会回退到默认值。也不建议配置成 `/`，因为 WebSocket 会用 `new URL("/api/ws", API_BASE_URL)` 拼接地址，基准地址需要是完整 origin。

## 4. 发布目录

示例目录：

```text
/var/www/flow-talk-web/build/client
```

把本地或 CI 构建后的 `build/client/` 上传到服务器对应目录即可。Nginx 的 `root` 应指向这个 `client` 目录，而不是项目根目录。

## 5. Nginx 配置

假设：

- 域名：`example.com`
- 前端静态目录：`/var/www/flow-talk-web/build/client`
- 后端监听：`127.0.0.1:8080`
- 后端 HTTP 接口前缀：`/api`
- 后端 WebSocket 入口：`/api/ws`

推荐配置：

```nginx
map $http_upgrade $connection_upgrade {
    default upgrade;
    '' close;
}

server {
    listen 80;
    server_name example.com;

    root /var/www/flow-talk-web/build/client;
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;
    }

    location = /api/ws {
        proxy_pass http://127.0.0.1:8080;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection $connection_upgrade;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_read_timeout 3600s;
        proxy_send_timeout 3600s;
    }

    location /api/ {
        proxy_pass http://127.0.0.1:8080;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

说明：

- `location /` 用 `try_files ... /index.html` 支持前端路由刷新。
- `location /api/` 代理普通 HTTP 接口。
- `location /api/ws` 代理 WebSocket，并保留 `Upgrade` 头。
- `proxy_pass` 不带 URI 部分，Nginx 会把原始 `/api/...` 路径原样转发给后端，避免尾部斜杠导致路径被意外改写。
- `map` 需要放在 Nginx `http` 作用域内，不能放到 `server` 里面。

## 6. 启用配置

检查 Nginx 配置：

```bash
nginx -t
```

重新加载：

```bash
nginx -s reload
```

如果服务器使用 systemd，也可以执行：

```bash
systemctl reload nginx
```

## 7. 验证

访问前端页面：

```text
http://example.com/
```

刷新前端路由，例如：

```text
http://example.com/login
```

验证后端代理：

```bash
curl -i http://example.com/api/users
```

如果返回 `401`，通常说明 Nginx 已经成功转发到后端，只是接口需要登录。

验证 WebSocket 地址：

```text
ws://example.com/api/ws
```

如果站点启用了 HTTPS，则使用：

```text
wss://example.com/api/ws
```

## 8. 常见问题

### 刷新页面返回 404

检查 Nginx 是否配置了：

```nginx
try_files $uri $uri/ /index.html;
```

### 前端请求到了 `127.0.0.1:8080`

说明构建产物里仍使用了本地开发默认地址。检查 `VITE_API_BASE_URL` 和 `app/request/index.ts` 的默认值，生产环境应使用同域地址。

### WebSocket 连接失败

重点检查：

- Nginx 是否配置了 `Upgrade` 和 `Connection` 头。
- `/api/ws` 是否转发到后端真实 WebSocket 地址。
- HTTPS 页面是否使用 `wss://`，不能使用 `ws://`。

### 接口 502

通常是 Nginx 连不上后端。检查后端服务是否启动，以及监听地址是否和 `proxy_pass` 一致。
