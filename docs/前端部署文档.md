# 前端部署文档

本文档说明 `flow-talk-web` 的生产部署方式。当前项目默认是 React Router SSR 构建，生产环境推荐使用 **Nginx + 前端 Node SSR 服务 + 后端 API 服务**：

```text
浏览器
  |
  v
Nginx :80/:443
  |-- /api、/api/ws  -> 后端服务 127.0.0.1:8080
  `-- 其他页面请求 -> 前端 SSR 服务 127.0.0.1:3000
```

当前项目开启了 SSR：

```ts
// react-router.config.ts
export default {
  ssr: true
};
```

SSR 模式不能只把 `build/client` 交给 Nginx 当静态站点托管。构建后会生成：

```text
build/client
build/server/index.js
```

其中：

- `build/client` 是浏览器静态资源。
- `build/server/index.js` 是 React Router SSR 服务端入口。
- 生产服务器必须启动一个 Node 进程运行 `build/server/index.js`。
- 项目已经提供启动脚本：`npm start`，实际执行 `react-router-serve ./build/server/index.js`。

只有明确不需要 SSR、想做纯静态 SPA 部署时，才把 `react-router.config.ts` 改成：

```ts
export default {
  ssr: false
};
```

本文默认按 SSR 部署说明。

## 1. 构建前检查

检查 Node 版本：

```bash
node -v
```

React Router 8 当前构建提示要求 Node 版本大于 `22.22.0`。如果低于该版本，可能暂时还能构建通过，但不建议用于生产构建环境。

安装依赖：

```bash
npm install
```

类型检查：

```bash
npm run typecheck
```

生产构建：

```bash
npm run build
```

构建完成后确认产物存在：

```text
build/client
build/server/index.js
```

浏览器端通过当前站点 origin 请求接口，也就是：

```text
/api/...
ws(s)://当前域名/api/ws
```

前端不再配置 `API_BASE_URL`，生产环境由 Nginx 把 `/api` 代理到后端服务。

## 2. 服务器目录

推荐部署到：

```text
/var/www/flow-talk-web
```

服务器目录至少需要这些文件：

```text
/var/www/flow-talk-web
  package.json
  package-lock.json
  build/
    client/
    server/index.js
  public/
```

如果在 CI 或本地已经完成构建，把这些文件上传到服务器即可。服务器上只运行生产依赖：

```bash
cd /var/www/flow-talk-web
npm ci --omit=dev
```

如果是在服务器上直接构建，则执行：

```bash
cd /var/www/flow-talk-web
npm ci
npm run build
npm prune --omit=dev
```

## 3. 手动启动 SSR 服务

进入部署目录：

```bash
cd /var/www/flow-talk-web
```

启动前端 SSR 服务：

```bash
NODE_ENV=production HOST=127.0.0.1 PORT=3000 npm start
```

等价于：

```bash
NODE_ENV=production HOST=127.0.0.1 PORT=3000 npx react-router-serve ./build/server/index.js
```

说明：

- `PORT=3000` 指定前端 SSR 服务监听端口。
- `HOST=127.0.0.1` 只允许本机访问 SSR 服务，由 Nginx 对外代理。
- 如果不设置 `PORT`，`react-router-serve` 默认会优先尝试 `3000`。
- `react-router-serve` 会同时处理 SSR 页面请求和 `build/client` 下的静态资源。

本机验证：

```bash
curl -I http://127.0.0.1:3000/
```

只要返回 `200`、`302` 或前端应用自己的响应，就说明 SSR 服务已启动。生产环境不要靠手动命令长期运行，建议用 systemd 或 PM2 守护。

## 4. 使用 systemd 守护 SSR 服务

创建服务文件：

```bash
sudo vim /etc/systemd/system/flow-talk-web.service
```

内容：

```ini
[Unit]
Description=Flow Talk Web SSR
After=network.target

[Service]
Type=simple
WorkingDirectory=/var/www/flow-talk-web
Environment=NODE_ENV=production
Environment=HOST=127.0.0.1
Environment=PORT=3000
ExecStart=/usr/bin/npm run start
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
```

启用并启动：

```bash
sudo systemctl daemon-reload
sudo systemctl enable flow-talk-web
sudo systemctl start flow-talk-web
```

查看状态和日志：

```bash
sudo systemctl status flow-talk-web
sudo journalctl -u flow-talk-web -f
```

如果服务器上的 `npm` 不在 `/usr/bin/npm`，先执行：

```bash
which npm
```

然后把 `ExecStart` 改成实际路径。

## 5. 使用 PM2 守护 SSR 服务

如果服务器已经使用 PM2，也可以这样启动：

```bash
cd /var/www/flow-talk-web
HOST=127.0.0.1 PORT=3000 NODE_ENV=production pm2 start npm --name flow-talk-web -- run start
pm2 save
```

查看日志：

```bash
pm2 logs flow-talk-web
```

systemd 和 PM2 选一种即可，不要同时用两套工具启动同一个端口。

## 6. Nginx 配置

假设：

- 域名：`example.com`
- 前端 SSR 服务：`127.0.0.1:3000`
- 后端服务：`127.0.0.1:8080`
- 后端 HTTP 接口前缀：`/api`
- 后端 WebSocket 入口：`/api/ws`

推荐配置：

```nginx
map $http_upgrade $connection_upgrade {
    default upgrade;
    '' close;
}

server {
    listen 80;
    server_name example.com;

    location = /api/ws {
        proxy_pass http://127.0.0.1:8080;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection $connection_upgrade;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_read_timeout 3600s;
        proxy_send_timeout 3600s;
    }

    location /api/ {
        proxy_pass http://127.0.0.1:8080;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

说明：

- SSR 模式下，`location /` 不能写成 `try_files $uri $uri/ /index.html`，而是要转发给前端 Node SSR 服务。
- `/api/` 必须放在 `location /` 前面，确保接口请求进入后端服务，而不是前端 SSR 服务。
- `/api/ws` 代理 WebSocket，并保留 `Upgrade` 头。
- `proxy_pass` 不带 URI 部分，Nginx 会把原始 `/api/...` 路径原样转发给后端，避免尾部斜杠导致路径被意外改写。
- `map` 需要放在 Nginx `http` 作用域内，不能放到 `server` 里面。

启用 HTTPS 时，把 `listen 80` 换成实际证书配置即可，核心代理规则不变。HTTPS 页面下 WebSocket 会使用：

```text
wss://example.com/api/ws
```

## 7. 启用配置

检查 Nginx 配置：

```bash
sudo nginx -t
```

重新加载：

```bash
sudo systemctl reload nginx
```

如果不是 systemd 管理 Nginx，也可以执行：

```bash
sudo nginx -s reload
```

## 8. 验证

验证前端 SSR 服务：

```bash
curl -I http://127.0.0.1:3000/
```

验证 Nginx 入口：

```bash
curl -I http://example.com/
```

刷新前端路由，例如：

```text
http://example.com/login
```

SSR 模式下这个请求应由前端 Node 服务返回页面，不能依赖 Nginx 的 `index.html` 回退。

验证后端代理：

```bash
curl -i http://example.com/api/users
```

如果返回 `401`，通常说明 Nginx 已经成功转发到后端，只是接口需要登录。

验证 WebSocket 地址：

```text
ws://example.com/api/ws
```

如果站点启用了 HTTPS，则使用：

```text
wss://example.com/api/ws
```

## 9. 更新发布流程

每次发版推荐按这个顺序：

```bash
cd /var/www/flow-talk-web
npm ci
npm run build
npm prune --omit=dev
sudo systemctl restart flow-talk-web
sudo systemctl reload nginx
```

如果构建在 CI 完成，则服务器只需要更新文件、安装生产依赖并重启 SSR 服务：

```bash
cd /var/www/flow-talk-web
npm ci --omit=dev
sudo systemctl restart flow-talk-web
```

Nginx 配置没有变化时，不需要每次 reload Nginx。

## 10. 纯静态 SPA 部署可选方案

如果不想在服务器上启动前端 Node 服务，可以关闭 SSR：

```ts
// react-router.config.ts
export default {
  ssr: false
};
```

然后重新构建：

```bash
npm run build
```

确认存在：

```text
build/client/index.html
```

此时才可以让 Nginx 直接托管 `build/client`：

```nginx
server {
    listen 80;
    server_name example.com;

    root /var/www/flow-talk-web/build/client;
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;
    }

    location /api/ {
        proxy_pass http://127.0.0.1:8080;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

注意：这是 SPA 部署，不是 SSR 部署。

## 11. 常见问题

### 页面请求返回 502

通常是 Nginx 连不上前端 SSR 服务。检查：

```bash
sudo systemctl status flow-talk-web
sudo journalctl -u flow-talk-web -f
curl -I http://127.0.0.1:3000/
```

### 接口请求返回 502

通常是 Nginx 连不上后端服务。检查后端服务是否启动，以及监听地址是否和 `proxy_pass http://127.0.0.1:8080` 一致。

### 刷新页面返回 404

SSR 部署时检查 `location /` 是否代理到了 `127.0.0.1:3000`。如果用了 `try_files ... /index.html`，说明配置成了 SPA 静态部署。

### 前端请求没有进入后端

检查 Nginx 是否配置了 `/api` 代理，并确认后端服务地址和 `proxy_pass` 一致。

当前代码逻辑是：

- HTTP 接口使用相对路径 `/api/...`。
- WebSocket 使用当前站点协议和域名连接 `/api/ws`。
- 前端不再读取或回退任何 `API_BASE_URL`。

### WebSocket 连接失败

重点检查：

- Nginx 是否配置了 `Upgrade` 和 `Connection` 头。
- `/api/ws` 是否转发到后端真实 WebSocket 地址。
- HTTPS 页面是否使用 `wss://`，不能使用 `ws://`。

### SSR 服务端请求后端地址怎么处理

浏览器端请求推荐走同域 `/api`，由 Nginx 转发到后端。SSR 服务端如果以后在 loader/action 中直接请求后端，不应该使用浏览器的 `window.location.origin`，而应该使用服务端专用后端地址，例如：

```text
http://127.0.0.1:8080
```

当前项目的页面数据主要在浏览器端请求；如果后续增加 SSR loader/action 请求，需要在代码里区分浏览器 API 地址和服务端 API 地址，避免把服务端内部地址打进浏览器构建产物。
